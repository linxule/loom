最近我做了一场题为 *Thinking Through AI*（经由 AI 思考）的线上讲座，由 Imperial / AOM OMT / AIIN New Scholar Series 主办，听众是三百多位研究者，九十分钟讲座，再加上一场很长的开放问答。那场讲座讲的是如何和 AI 智能体一起研读学术文献——一条介于全盘托付与主动弃绝之间的第三条路。（录像和幻灯片在[我的讲座页面](https://linxule.com/talks/thinking-through-ai-01)；Kevin 和我写的 *[Interpretive Orchestration](https://doi.org/10.1177/14761270261448645)* 论文前一天刚发表，那篇是更长的配套读物。）

在那九十分钟里的某个时刻，我开始注意到一个模式。有人不断举手。聊天窗口也满了。到最后，打字提问和口头提问加起来大概三十五个。看上去，它们似乎指向不同的方向。

Mallika 的提问很坦率，说出了很多人都感受过的困境：

> "像这次讲座里的这些论文，我无从知道每一篇的具体内容。"

她把五十篇论文交给 [Claude Code](https://www.claude.com/claude-code) 跑；智能体到三十篇之后开始偷懒；她把剩下的转到高推理力度的 Codex，结果拿回来五条编造出来的主张。Veroniek 问的是同一个问题，只是穿上了覆盖范围的外衣——一篇普通的 AMJ 或 SMJ 论文，会不会出现在深度研究流程里？Samrat 问的是同一个问题，只是穿上了治理的外衣——我们把付费墙后的 PDF 喂给生成式 AI 平台时，是不是违反了出版商合同？Sara 三次追问的也是同一个问题，只是换成了“我该从哪里开始”——问得不躲不闪，却也没有立足点。Jeannel 问的是同一个问题，只是穿上了教学困境的外衣——她所在的写作中心说不能用 AI，而我的讲座说先试试看再决定，那她星期一到底该怎么办？Salih 在聊天区现场做了一个实验来提出这个问题：他把同一个主题下模型自己的草稿和一篇已发表的学术引言交给模型比较，模型偏向了自己。

把这些外衣剥掉，问题其实归结为一个。*当我不可能亲自读完所有东西时，我到底该在智能体协作中扮演什么角色？我们如何负责任地提出并捍卫那些借助 AI 智能体获得的洞见？*

九十分钟装不下我想给出的答案。这是更长的版本，也是我欠 Mallika 和那场问答里其他人的回答。

先声明：这是我现在能给出的阶段性答案。它足够确切，值得拿出来分享；也足够不确定，我希望你们来挑战它。它由三句话组成：

> *提示词就是洞见。参与过程就是技艺。对话记录就是产物。*

每一句都对应同一个分析过程里的一个时间阶段——开始提问之前你做什么，工作与思考正在发生时你做什么，结束之后你留下什么。每一句都是我试图在讲座中演示的东西，也分别对应现场不断抛给我的不同面貌的担忧。我想带着这些担忧，把它们逐一走一遍。

# 提示词变成工作的时候

我在讲座里、也在真实生活中，都是从这里开始的。我手里有两篇正在读的论文，一个直觉，还有一个开场问题。*当管理学研究者对大语言模型做审计或实验时，他们使用什么方法，又提出什么类型的主张？* 这是个合理的问题。和 Claude 对话到第五轮，提示词才终于成型——而且只是因为我让它变成了另一个问题：*当管理学研究者把大语言模型当作理论研究对象，追问它们作为评估者、决策者、推理者到底意味着什么时，他们拿什么证据来支撑这些主张？*

我想为之命名的，正是这个转折。这里的“工作”不是评判深度研究报告的质量，而是打磨提示词。为了写出这个提示词而进行的五轮对话，就是方法论本身——在这一部分里，我和 Claude 一起弄清楚，我真正要找的到底是什么。等提示词足够好时，文献搜索几乎成了顺带的事。

我觉得，这恰恰是最容易被跳过、也最容易被一语带过的部分（好像有经验的学者天生就知道，面对任何一批文献该问什么问题）。模板听起来像是能帮上忙。大家不断向我要一个。我没有可以给的模板，因为模板会代替提示词本来应该完成的事情。通过来回对话获得的清晰度，才是“交付物”。

在讲座里，我把三种配置称为进入 AI 智能体工作的上手路径（不是等级序列）。Sara 问从哪里开始；Peter 问他大学的 Microsoft Copilot 许可够不够。诚实的回答是，配置一——打开 Claude.ai（或类似工具），放入三篇论文，提出一个问题——已经足够完成“提示词就是洞见”这句话所指的工作。你可以在三十分钟之内进行一场富有成效的研究对话。配置二启动一条深度研究流程，让智能体向外扩展、跑上一个小时。配置三进入终端，让每一次工具调用都摊开在你面前。也许有点反直觉：没有哪一种*更好*。它们的区别在于，AI 智能体的编排过程有多少被显露出来。

我自己这样做的时候，搭建那套环境主要是为了让提示词在一切同时发生时仍然可以修改。智能体要读的东西全部转成 Markdown，因为 PDF 对模型来说是图像，Word 则是纯文本加格式噪音。论文来自我自己的 Zotero 文献集合，导出到一个沙盒文件夹：智能体可以读，但不能向沙盒外写入。桌面上的模型取决于哪家实验室那天状态好——多数日子用 Claude，验证用 Codex，第二意见用 [Kimi](https://www.kimi.com)。Sangita 问延迟和 token 消耗；我保留三个账号（Anthropic、OpenAI、Moonshot），按项目需要分流，并把 [OpenRouter](https://openrouter.ai) 设为后备，这样单个实验室出问题不会让工作停摆。这些都不算什么稀奇配置。它只是一套基础设施，确保提示词在你已经动起来的时候仍然可以修改。

对于还完全没做过这些的同事，我做了 [Carrel](https://github.com/linxule/carrel)：一个 Claude Code 插件，会引导 Claude 询问你、核查你的机器（macOS、Linux、Windows），然后安装并配置其余部分。你只需要回答问题，基本不用碰终端（大体如此——不过要让 Claude 或 Codex 在自己机器上跑起来，还是需要用到一点终端）。最终的配置*本身*有点像一种个人化的教学法：研究者在选择自己需要什么的过程中，也学会这些工具到底做什么。对于那些想要一个“AI 零基础”式上手指南、但又不想被贴这个标签的人来说，这就是。智能体式 AI 驾驭工具的插件生态正在快速增长，在高层次上也正在收敛。所以你大概也可以直接请 [Codex](https://openai.com/codex/) 帮你安装插件，让它自己处理差异，把这套东西改到适合你在 Codex 桌面应用和 CLI 里的工作流。

> 对看过线上讲座的人来说，子智能体、MCP 服务器和智能体群都是锦上添花——一开始不需要它们。配置一已经足够。

如果想看更深的地图——Zotero、Research Rabbit、[Obsidian](https://obsidian.md)、MCP 配置——目的地是 [research-memex.org](https://research-memex.org/introduction/getting-oriented)。

Li Jiang 问了 Kimi 智能体安全，但我们没来得及展开。简短回答是，[kimi-plugin-cc](https://github.com/linxule/kimi-plugin-cc) 里的边界不在系统提示词。它是一份白名单（可以理解成访客名单），由伴随运行时（可以理解成门口的保安）在任何写入或命令行指令真正运行之前执行：能识别符号链接的路径边界检查、排除 `.git/`、并明确拒绝包管理器触发的不透明脚本。在只读模式（`/kimi:ask`、`/kimi:review`、`/kimi:challenge`）里，智能体完全不能写。规则不是放在提示词里。运行时才是边界。

# 当模型意见不一致

Mallika 的案例是我一直回头想的那个。五十篇论文，Claude Code 到三十篇后开始偷懒，高推理力度的 Codex 产出五条编造主张。直觉上，我们会问该相信哪个模型。这个问题问错了。真正的问题是：当两个模型读同一篇论文，却得出不同结论时，你做什么？

两个模型读同一条引用却得出不同结论，这正是我们想要呈现出来、并且*值得深入探究*的分歧。有时候，一个对，一个错。有时候，两者都部分正确。有时候，源 PDF 里的引用格式本身有误，而分歧正是提醒你手动重查的信号。分布式并行读取的意义——在每条分析流里用两到三个模型厂商，把它们当作平行读者，而不是冗余副本——就在于分歧才是信号。如果它们全都同意，你学到的是：一个听起来很自信的主张，也被基于重叠语料训练出来的模型广泛持有。这不是毫无意义，但我不会把它看作验证。也许它更像是在告诉你，文献中的共识大概落在哪里。

验证是下一步，而且需要刻意带上对抗性。一旦有了综合稿，我会派子智能体去检查那些可测试的主张——也就是带引用的主张——让它们去读真正的论文并核对主张（不是总结，也不是转述）。这一步本来会抓住 Mallika 那五条编造主张——不是把它们当成结论收下，而是先归入“分歧”。这里第二轮的意义不只是准确性检查，还触及对被引用主张背后假设的理解，以及它们是否能与综合稿里的假设兼容（想想边界条件和认识论差异）。

在模型与模型之间的分歧之下，还有更深一层的分歧——机器读到的东西与你读到的东西之间的分歧。[interpretive-orchestration](https://github.com/linxule/interpretive-orchestration)（另一个 Claude Code 插件）将一种让这一区分持续可见的方法固化成了代码：三明治方法——先人工编码，再 AI 协作编码，最后人工综合——并且有一个钩子会在第一阶段完成之前*阻止*第二阶段开始。跳过人工编码，智能体就拒绝启动。这个钩子，就是被编译成代码的论证担保。迈向让产物本身成为论证的更大一步，写在 [LOOM XV: Theorizing by Building](https://www.threadcounts.org/p/loom-xv-theorizing-by-building-018) 里。

Salih 在聊天区做的实验，其实是现场众人在不知不觉中完成的一次诊断。给模型看它自己的草稿，以及同一主题上一段已发表的学术引言，模型偏向了自己的文字。稳定地偏向。Nan Hu 曾试探性地提出，可以把 AI 当作筛稿评审，用来应对 Lamar 的 *More Versus Better* 所描述的发表洪流。Salih 的发现，正是射向这个提议的那枚鱼雷。一个偏爱长得像自己的东西的模型，如果被部署成守门人，就会筛选出长得像自己的东西。那道门会把人的写作挡在门外。

Jeannel 的困境也有同样的结构。她所在的写作中心处境艰难，因为摆在它面前的那个二元对立本身就是假的。禁止 AI，你就把主动权让出去。允许 AI 但没有方法论，你就只是让那股洪流更快。第三条路是一种姿态，而“参与过程就是技艺”所要命名的，正是这种姿态。2025 年 9 月，面对 Imperial 新入学的博士生群体（严格说是 MRes——英国的博士项目从研究型硕士这一年开始），我们在 Claude Code 里一起搭建智能体式工作流，使用不同模型、分配不同角色（可以在[这里](https://research-memex.org/case-studies/systematic-reviews/systematic-review-syllabus)读更多）。当我们一步一步组装编排流程时，学生看见自己的问题框定被智能体的输出反弹回来，也学会去修改*那个问题框定*。这才是我希望写作中心教的东西，而不是把重点放在最佳提示词/配置/工具上，试图产出“最佳”输出（对社会科学里的许多分析任务来说，这些东西很难确定，也很难验证）。

Youngbeom 在聊天里反驳了这种编排方法，引用 Karpathy 的说法：不要用僵硬的人类语言工作流约束智能体；给出成功标准（理想情况下是一个 Python 验证器）和一个可运行示例，让模型自己找路径。这种方法在编程，以及那些判断标准可以被清楚指定、解空间有边界的领域里确实很好用。但是，在诠释性研究/任务里，没有一个 `pytest test_literature_review_is_good`——当没有别的东西能承载标准时，提示词就必须*承载*标准。所以，没错，我们确实是在坚持以人的能动性为中心：我是编排者，而我的编排提示词是密集、多阶段的。Kevin 和我写的 *[Interpretive Orchestration](https://doi.org/10.1177/14761270261448645)* 论文对这条思路展开得更多。

# 窗口之后留下什么

这一部分，是现场问题变得最锋利、而现场回答最薄的地方。于是，我在这里补上。

*Veroniek 关于覆盖范围的问题。* 这是不是意味着论文必须开放获取，智能体才看得到？一篇普通的 AMJ 或 SMJ 论文，会出现在深度研究流程里吗？

诚实回答：今天生产环境里的深度研究智能体会爬取它们能在开放网络上获取到的内容。它们会系统性地漏掉顶尖的管理学期刊，因为这些期刊在付费墙后面。它们会用摘要、二手介绍、会议版本、预印本，以及它们可能在训练中见过的东西来补偿。我们拿回来的地图可能偏向更知名的学者和期刊，书目也可能偏向付费墙之外的东西。实际做法有两个。第一，从你所在机构有授权访问权限的 Zotero/EndNote 文献集合导出相关论文，把 PDF 转成 Markdown，再让智能体指向这些 Markdown 文件。第二，把任何深度研究输出都当作*检索辅助*，而不是完整书目——然后在旁边同时跑传统的 Web of Science 或 Scopus 检索（给你的关键词和搜索标准做版本记录！）。Iikka 在聊天里分享了一个 [Elsevier 研究者服务](https://researcher.elsevier.com/eur/) 链接，指向的是同一个方向。这类服务也许最终会填上这个缺口，但我会围绕当前限制做计划，而不是希望它自己消失（反正那很可能只是意味着要多付订阅费）。

*Samrat 关于治理的问题。* 我们把付费墙后的 PDF 上传到生成式 AI 平台时，是不是违反出版商合同？

诚实回答：这是一个灰色地带，大多数大学法务还没有真正厘清。我的工作原则有三部分，而且都围绕窗口之后留下什么。把机器可读副本留在本地——PDF 转 Markdown *可以* 在本地完成，Markdown 放在项目沙盒里。不要把 PDF 上传到你无法控制的聊天界面——智能体从本地沙盒读取；需要的话选择不参与训练。在项目记录里记录服务商路由——调用过的每一个模型、发起调用的每一个账号，都要注明日期、确保可追溯。这条线到底划在哪里，取决于你的 IRB 和你的学科。不要假装这条线不存在。

*Daniel 问了 PDF 转 Markdown 这一步。*

提醒一句：转换不是无损的。容易在本地机器上跑起来的工具，仍然比不上云端运行的工具。表格、数学符号、脚注、精心排版——都会退化。如果智能体读到的是一张乱码表格，综合稿从源头上就被污染了。[MinerU](https://www.mineru.org/) 处理高难度情形（复杂表格和图形上 90%+）；markitdown 处理日常情形；两者都打包在 Carrel 里。当保真度重要时，先检查转换后的文件，再相信智能体对它的阅读。

*运行时那个回答漏掉的是数据驻留。*

Kimi 跑在 Moonshot 位于中国的基础设施上。如果你的研究使用敏感数据（访谈记录、未发表草稿，任何受 IRB 约束的东西），就要有意选择服务商，并记录这个选择（例如，通过 OpenRouter 使用由欧盟、北美节点提供服务的开源模型）。

在这些问题之下，我认为真正的“留下什么”之问正是这个——而 [memex-plugin](https://github.com/linxule/memex-plugin)（适用于任何聊天机器人或智能体）试图给出的，正是针对这一问题的可落地方案。长项目会跨越会话、模型、压缩窗口——单个智能体能放在工作记忆里的东西很小，而你之后真正想引用的东西更大。Memex 留下来的东西分两层。第一层备忘录由那个*当时在场*的智能体在对话还活着的时候写出。第二层备忘录是之后从记录中重构出来的。它们不是同一种产物。第一层备忘录承载的是亲历过那项工作的重量。第二层备忘录是对它的一次合格的阅读。两者都有用。差异在你引用时很重要。Anja 的三角互证担忧、Hovig 的“痕迹是 1:1 吗？”这个问题，以及 Mallika 那句“我无从知道”的隐含版本，都归结于此。跨会话留下、以后可以被引用的到底是什么？它的保真度又到什么程度？

披露标准是所有这些问题的下游，而且它正在到来。我预计编辑很快会要求 AI 使用附录。标准还没有稳定下来。我现在的做法是：在方法附录里单列一节，说明模型和版本，链接相关编排提示词（必要时匿名化），并为每个分析阶段放入一个有代表性的脱敏记录。Memex 捕捉原始材料；附录是被策展过的子集。

问题之下还有问题。我始终回到的是这个：

> *什么时候，经由 AI 中介的阅读才算阅读？*

当一个 Kimi 智能体群处理五十篇论文并产出综合稿时，在博士资格考试和终身教职评审仍然认可的那种“阅读”意义上，这位学者是否真的读过这批文献？Memex 的第一层 / 第二层区分在代码里做了其中一部分工作——那个亲历了对话的智能体，写出的备忘录和后来从记录中重构出来的备忘录不是同一种东西。这是操作层面的答案。认识论层面的答案仍未稳定，而且我不觉得那些我最想一起思考这个问题的人已经有了答案。

还有几个问题我这里只点到为止。供应商选择的政治经济学——我们不会接受单一出版商拥有引文索引，那为什么要接受单一供应商掌控 AI 中介阅读的入口？训练数据污染——模型是在我们的草稿和审稿报告上训练的，而当我们用它们来综合关于自己领域的文献时，认识论回路会进一步收紧。公平与双层准入——一套只有在有闲钱花在 API 额度上时才能扩展的工具栈，是一种特权；我们若把它称为方法论，未免太不明智。

还有一件关于现场的事。当我在讲的时候，聊天区正在做这场讲座所讲的事。Susan Lanz 主动推荐了 NotebookLM，因为它的引用可以链接回原文。Shawna Calhoun 分享说，她会让 AI 阅读器把论文念出来，自己一边听一边读——朗读帮助她捕捉 AI 的主张哪里需要澄清或修正。Sean Sullivan 丢出了 [lit-lake](https://github.com/ElliotRoe/lit-lake)，一个 Zotero-Claude 桥接工具。Iikka 分享了 Elsevier 服务。Salih 自己在现场跑了那个实验，并把结果贴出来。这篇文章试图命名的东西，已经在现场发生了。*参与过程就是技艺。* 技艺是分布式的。
